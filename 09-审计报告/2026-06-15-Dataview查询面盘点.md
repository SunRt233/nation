# 2026-06-15 Dataview 查询面盘点

> 本报告只盘点“会影响当前系统页口径、Agent 入口链路、任务卡聚合、工作日志聚合”的查询面。
> 不试图逐页整治全库所有内容页里的局部 Dataview 关联块。

## 一、结论先看

- 当前**没有发现必须立刻修的主链路查询故障**。
- 这轮 `type` 收口后，`00-首页` 与 `02-数据库` 的核心查询面已经基本跟上。
- 当前更像是“还有少量旧思路残留，建议继续收口”，而不是“查询层已经影响 Agent 正常工作”。

## 三、核心查询面现状

### A. 已和新口径一致，可继续作为真相源

以下页面/数据库已经直接使用新口径，当前无需改动：

- [[00-首页/活跃任务]]
  - 统一查询 `FROM "00-首页/活跃任务"`
  - 统一筛 `WHERE type = "活跃任务卡"`
- [[00-首页/系统页角色索引]]
  - 统一筛 `WHERE type = "系统"`
  - 用 `role` 做细分分层
- [[00-首页/自动汇总/自动汇总-任务卡健康检查]]
  - 与任务卡体系一致，口径稳定
- [[00-首页/自动汇总/自动汇总-异常巡检]]
  - 主要做中间层扫描与提示，不依赖旧 `type`
- [[02-数据库/活跃任务.base]]
  - 过滤条件为 `file.inFolder("00-首页/活跃任务") + type == "活跃任务卡"`
- [[02-数据库/工作流.base]]
  - 当前按目录聚合 `00-首页/工作日志`
- [[02-数据库/数据库-工作流]]
  - 任务区块已切到 `活跃任务卡`

### B. 建议继续收口，但不影响当前运行

以下页面能跑，也不会立刻拖垮 Agent，但还带有“旧信息架构思路”，建议后续继续整理：

- [[00-首页/自动汇总/自动汇总-最近任务摘要]]
  - 现在仍通过扫描 `00-首页/工作日志/` 下 `## 📅 YYYY-MM-DD ...` 标题来抽“最近任务”
  - 这不是错，但仍偏旧思路：它抽的是“日志段标题”，不是“任务卡事实”
  - 风险：如果日志标题写法漂移，摘要会失真
- [[00-首页/状态摘要]]
  - 现在对“最近任务”仍嵌入上面的日志标题抽取结果
  - 因此状态页的“最近任务”模块，仍间接受日志正文格式约束
- [[02-数据库/数据库-工作流]]
  - 一部分任务视角已经迁到 `活跃任务卡`
  - 但日志相关 DataviewJS 仍是“扫描日志正文结构”的思路，而不是纯 frontmatter / 任务卡驱动

### C. 可保留，不建议为了统一而硬改

以下 Dataview 页面主要是内容页里的局部关联查询，和这次系统口径治理不是同一层问题，可保留：

- `03-知识点/` 下的大量“关联知识点 / 相关专题 / 延伸阅读”查询
- `02-考纲条目/` 下的大量索引型查询
- `04-专题与题型/`、`04-题库/` 下的内容侧关联查询
- `07-资料提炼/`、`10-索引与统计/` 中的局部聚合页

> 这些页面多数依赖的是 `知识点 / 题目 / 路径范围 / tags / syllabus_codes` 等内容层字段，不是这次 `系统 / 活跃任务卡 / 工作日志` 收口的风险点。

## 四、当前还在用“旧思路”的地方

这里的“旧思路”不是指查询报错，而是指应用层仍沿用旧数据组织方式。

### 1. 日志仍承担“最近任务摘要”中间源

涉及页面：

- [[00-首页/自动汇总/自动汇总-最近任务摘要]]
- [[00-首页/状态摘要]]

表现：

- 通过扫描 `00-首页/工作日志/` 的标题来表示“最近做了什么”
- 语义上更接近“最近日志条目”，而不是“最近任务事实”

影响：

- **不会影响 Agent 主链路启动**
- 但会让“最近任务”模块继续依赖日志书写格式，而不是结构化任务卡体系

建议：

- 若后续想增强自动化，可考虑把“最近任务摘要”逐步改成：
  - 优先读 `活跃任务卡`
  - 再辅以日志证据链接

### 2. 工作流页仍混合“任务卡视角”和“日志正文视角”

涉及页面：

- [[02-数据库/数据库-工作流]]
- [[02-数据库/工作流.base]]

表现：

- 任务面已切到任务卡体系
- 日志面仍按目录和正文结构工作

影响：

- 当前合理，不算错误
- 但从架构纯度上看，这是一种“过渡态混合”

建议：

- 保留 `工作日志` 作为证据与过程层
- 不要再让它承担短期任务真相源
- 若未来继续升级，可把“工作流”页写得更明确：
  - 任务事实看任务卡
  - 过程证据看日志

## 五、对 Agent 的实际影响判断

### 现在会不会影响 Agent 工作

不会明显影响。

原因：

- `状态摘要 → 活跃任务 → Agent战略方向` 主链路是通的
- 任务聚合页和任务数据库都已切到 `活跃任务卡`
- 系统页角色索引已切到 `type = "系统" + role`
- Obsidian 内部实时渲染验证也已通过

### 如果不继续收口，会有什么中长期影响

- “最近任务”类展示还会继续受日志写法影响
- 后续若继续做更强自动汇总，任务事实和日志事实会继续并存
- 将来做更细粒度的 Agent 路由、自动审计、统一仪表板时，需要额外兼容这层过渡逻辑

## 七、建议的下一步

如果继续做，我建议顺序是：

1. 先不碰全库内容页
2. 只补 `最近任务摘要` 这一层中间事实
3. 再决定是否把 `状态摘要` 的“最近任务”模块一起切换

这样收益最大，也最不容易引入新的展示回归。
