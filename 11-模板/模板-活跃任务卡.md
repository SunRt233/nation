# 模板-活跃任务卡

> **用途**：为 `00-首页/活跃任务/` 下的单个任务卡提供统一 frontmatter 与正文结构。
> **原则**：
> - 任务卡是真相源：记录“还有什么没做 / 为什么重要 / 证据在哪”
> - `[[活跃任务]]` 只负责聚合展示，不手写维护整份任务清单
> - `[[工作日志]]` 只记录过程、验证、决策依据，不替代任务卡状态

## Frontmatter 模板

```yaml
---
title: {任务标题}
type: 活跃任务卡
status: active
priority: P1
area: {领域}
owner: Agent
created: YYYY-MM-DD
updated: YYYY-MM-DD
completed:
source_notes: [[来源笔记]]
related_notes: [[相关笔记1]], [[相关笔记2]]
evidence: [[工作日志/证据1]], [[工作日志/证据2]]
---
```

## 字段说明

| 字段 | 必填 | 含义 | 取值建议 |
|---|---|---|---|
| `title` | 是 | 任务名 | 用“动作 + 对象”命名 |
| `type` | 是 | 文件类型 | 固定为 `活跃任务卡` |
| `status` | 是 | 当前状态 | `active` / `completed` / `blocked` |
| `priority` | 是 | 优先级 | `P1` / `P2` / `P3` |
| `area` | 是 | 任务所属领域 | 如：备课体系 / 知识点维护 / 脚本维护 |
| `owner` | 是 | 主要责任方 | `Agent` / `User` / `Agent+User` |
| `created` | 是 | 建卡日期 | `YYYY-MM-DD` |
| `updated` | 是 | 最近更新时间 | 每次推进都更新 |
| `completed` | 否 | 完成日期 | 仅 `status: completed` 时填写 |
| `source_notes` | 是 | 任务来源 | 触发此任务的入口文件 |
| `related_notes` | 是 | 相关资料 | 专题页 / KP / 模板 / 战略页等 |
| `evidence` | 是 | 证据链接 | 日志、产出文件、验证文件 |

## 正文模板

```md
# {任务标题}

## 目标

一句话说明这张任务卡要完成什么。

## 完成标准

- [ ] 标准 1
- [ ] 标准 2
- [ ] 标准 3

## 当前判断

- 为什么现在要做
- 当前卡点是什么
- 若与长期方向有关，具体关联到哪里

## 下一步

1. 下一步动作
2. 下一步动作

## 备注

- 可选：补充上下文、约束、待确认事项
```

## 更新规则

1. 新任务：直接复制本模板，新建到 `00-首页/活跃任务/`
2. 推进中：更新 `updated`，必要时补充“当前判断 / 下一步”
3. 已完成：把 `status` 改为 `completed`，填写 `completed`
4. 被阻塞：把 `status` 改为 `blocked`，并在正文写清阻塞原因
5. 不删除旧任务卡：历史任务保留，用于回溯与审计

## 最小示例

```md
---
title: 断链检测脚本 aliases 支持
type: 活跃任务卡
status: active

owner: Agent

completed:
source_notes: [[状态摘要]]

---

# 断链检测脚本 aliases 支持

## 目标

让断链检测脚本识别 frontmatter aliases，减少假阳性。

## 完成标准

- [ ] 脚本读取 aliases
- [ ] 审计结果减少已知假阳性
- [ ] 工作日志记录验证结论
```
