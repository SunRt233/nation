# 课件加工页

> 本目录承接“知识库内已经基本成型的 Markdown 教学内容”到最终交付物 `.docx / .pptx` 的最后一跳。
> 它不替代 `备课大纲/`、`新授课/`、`专题页`，而是把这些上游内容压缩成一张**可执行加工单**，供 Agent 或脚本继续产出正式课件。

> **页面边界（2026-06-16）**：本页负责“什么时候该继续加工、加工时读哪些源页、产出哪些文件、做到什么程度算完成”。
> - 备课逻辑、认知台阶、媒介分工：仍以 `[[04-课件/备课大纲/README]]` 与具体备课大纲为准
> - 学生可见正文：仍以 `[[04-课件/新授课/README]]` 与具体讲义页为准
> - 字体、公式、导出脚本等技术细节：去 [[11-模板/模板-课件产出规范]]

## 公式与导出策略（2026-06-16 更新）

当前 `.docx` 加工默认采用以下顺序：

1. 关键公式优先使用 Word 原生数学对象（OMML）
2. 普通短化学式、简单上下标优先用正常文本排版
3. 只有原生公式明显不够用时，才使用公式图片兜底

这样做的原因是：

- 比手工 PNG / SVG 摆位更接近教材和 Word 原生公式效果
- 上下标、分式、根号、括号间距更稳定
- 教师稿、学生讲义、PPT 后续更容易复用同一套公式资产
- 文件体积通常也比“大量公式图片化”更可控

### 实务口径

- 教师稿、学生讲义中的核心展示公式：默认做成 OMML
- 表格中的短公式、简短变形式：可保留文本
- 化学式尽量规范成真实下标，如 `H₂O`、`CO₂`、`NH₃`、`Ca(OH)₂`
- 面向学生的正文里，尽量避免长期保留 `E_k`、`ΔT_b` 这类下划线式记号

### 导出验收

`.docx` 产出不以“脚本成功运行”为完成，而以“可视化渲染检查通过”为完成。默认验收顺序：

1. 生成 `.docx`
2. render 成页面图片或 PDF
3. 逐页检查公式、表格、分页、页眉页脚
4. 有问题则回脚本修正，再次导出

### Windows 渲染兼容补丁（2026-06-16）

Windows 环境下，若上游通用 `render_docx.py` 因 `LibreOffice` 本地 profile 路径写法或 `pdf2image/poppler` 缺失而失败，当前工作流默认改走本地包装器：

- 脚本位置：`11-模板/scripts/render_docx_windows.py`
- 一键入口：`11-模板/scripts/render-docx.ps1`
- 超短入口：`11-模板/scripts/render-docx.cmd`
- 适用场景：本机安装了 `LibreOffice`，但通用渲染脚本在 Windows 上无法稳定产出 PNG
- 兼容策略：
  - 用 `Path(...).as_uri()` 生成 Windows 可识别的 `file:///C:/...` profile URI
  - 直接定位 `soffice.exe` 绝对路径，而不是依赖当前 shell 的 PATH 刷新
  - 用 `pypdfium2` 栅格化 PDF，绕过 `pdf2image` 对 `poppler` 的额外依赖
  - 包装器脚本保持自足，不再绑定插件缓存目录里的特定 `render_docx.py` 版本路径

Windows 下的推荐决策顺序：

1. 先尝试通用 `render_docx.py`
2. 若失败原因为 `LibreOffice profile`、`soffice PATH`、`poppler/pdf2image`，立即切到 `render_docx_windows.py`
3. 验收仍以“成功产出 PDF/PNG 且逐页目检正常”为准，不以 stderr 是否完全安静为准

推荐命令：

```powershell
& 'C:\Obsidion\妙妙屋\11-模板\scripts\render-docx.ps1' `
  'C:\Obsidion\妙妙屋\06-学生侧材料\讲义\2026-06-16-气体与溶液-基础班-学生讲义.docx'
```

若要更短，也可直接：

```powershell
C:\Obsidion\妙妙屋\11-模板\scripts\render-docx.cmd "C:\Obsidion\妙妙屋\06-学生侧材料\讲义\2026-06-16-气体与溶液-基础班-学生讲义.docx"
```

默认行为：

- 不写输出目录时，自动输出到输入文件同目录下的 `<文件名>_render/`
- `render-docx.cmd` 默认附带 `-EmitPdf`
- 适合直接拖拽 `.docx` 到 `.cmd` 上执行

口径上，这个包装器属于“知识库工作流层补丁”，优先保证本地导出链路可用；若未来插件侧原脚本修复，可再评估是否退回统一入口。

## 为什么要单独加这一层

过去已经有过 `.docx / .pptx` 试点产出，也有 `[[模板-课件大纲]]` 与 `[[模板-课件产出规范]]`，但仍有一个空档：

1. 上游 Markdown 已经能上课，不等于已经适合直接导出
2. Agent 容易跳过“要不要继续做成 docx/pptx”的判断
3. 同一主题可能同时需要教师稿、学生讲义、投影 PPT、打印版题单，但抽料来源不同
4. 缺少一张中间页时，试点产物、脚本、回写位置很难统一

所以这里新增“加工页”这一层，专门回答：

- 这次到底要产出 `docx`、`pptx`，还是只停留在 Markdown
- 每种产出分别以哪几页为真相源
- 哪些内容适合进 PPT，哪些必须留在板书或讲义
- 成品落到哪里，验证后该回写哪里

## 默认链路

```text
知识点 / 专题页 / 教学洞察
  ↓
备课大纲（教师设计）
  ↓
新授课 / 习题课 / 复习课（Markdown 成稿）
  ↓
加工页（本目录，决定产出范围与抽料规则）
  ↓
试点产出 / PPT / 学生讲义目录中的 .docx / .pptx
```

## 什么时候需要建加工页

满足任一情况就建议建：

1. 用户明确说“转成 Word / PPT / 可打印讲义 / 正式课件”
2. 需要把同一份 Markdown 同时加工成多个交付物
3. 需要明确区分“PPT 展示内容”与“板书/讲义保留内容”
4. 需要给后续 Agent 留一张稳定的加工说明，而不是让它每次重读整份讲义自由发挥

## 什么时候不必建

以下情况先不要建加工页：

1. 上游备课大纲或讲义还处于明显草稿态
2. 还没明确是要教师稿、学生讲义还是投影课件
3. 只是要补教学图、真题链、配套题单，不涉及正式导出

## 加工页的最小职责

每张加工页至少要回答 6 件事：

1. `source_of_truth`：这次抽料的真相源是哪几页
2. `deliverables`：要产出的文件清单是什么
3. `audience_split`：教师 / 学生 / 投影分别看什么
4. `media_split`：哪些内容进 PPT，哪些保留板书，哪些留在讲义
5. `output_paths`：最终落点目录与命名
6. `done_definition`：做到什么程度算完成

## 推荐目录与落点

| 产出物 | 推荐落点 | 说明 |
|:---|:---|:---|
| 教师上课大纲 `.docx` | `04-课件/试点产出/` | 先在试点目录验证样式 |
| 正式投影课件 `.pptx` | `04-课件/试点产出/` 或 `04-课件/PPT/` | 试点稳定后可迁到 `PPT/` |
| 学生讲义 `.docx` | `06-学生侧材料/讲义/` | 作为学生交付物管理 |
| 可打印题单 / 便携卡 `.docx` | `06-学生侧材料/讲义/` 或对应子目录 | 以学生材料目录为主 |
